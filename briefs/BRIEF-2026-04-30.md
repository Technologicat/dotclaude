## 1. Goal & non-goals

**Goal.** Put the Claude Code (CC) configuration directory `~/.claude/` under git, hosted on GitHub, so that changes can be synced between dev machines via a simple `git pull`. Trim `CLAUDE.md` along the way to reduce attention dilution on long-running sessions.

**Non-goals.**
- Not syncing runtime state (conversation history, sessions, tasks, telemetry). Each machine keeps its own.
- Not syncing auto-memory (`memory/`). Each machine accumulates independently; high-signal observations are promoted to `CLAUDE.md` or to `substrate-independent/field-observations.md` via the existing ethnography pipeline. Known tradeoff: a CC session on one machine won't know what a CC session on the other learned yesterday.
- Not unifying `CLAUDE.md` and `CLAUDE_webchat.md` content. They serve different surfaces (CC vs claude.ai) and prefer different wordings — `CLAUDE_webchat.md` is a verbatim local backup of the user's claude.ai userPreferences.

## 2. Repo layout

`~/.claude/` itself is the git repository. One `git pull` updates everything; no symlink dance.

The challenge: CC continuously writes runtime state into the same directory. The defense is a **whitelist `.gitignore`** — ignore everything at the root by default, then explicitly opt in to tracked items. New CC state files/dirs in future releases are gitignored automatically; tracking them requires an explicit decision. **This is by design** — when a new directory appears in `git status --ignored` after a CC update, that's the safety net working, not a problem.

### Files & directories to track

Top-level files:
- `.gitignore`, `.gitattributes`
- `README.md` (new — orientation for GitHub visitors)
- `CLAUDE.md` — global CC instructions (will be trimmed; see §5)
- `CLAUDE_webchat.md` — verbatim claude.ai userPreferences backup
- `NEW-MACHINE-SETUP.md` — sysadmin checklist
- `VKBASALT-SETUP.md` — gaming postprocessing setup (referenced from NEW-MACHINE-SETUP.md; travels with it)
- `settings.json` — CC settings (permissions allowlist, statusline, effortLevel)
- `statusline.sh` — custom CC status line

Top-level directories:
- `commands/` — slash commands (currently empty; `callgraph` was converted to a skill on 2026-06-05 — see §6. Directory kept in the whitelist for future commands.)
- `scripts/` — utility scripts. Will hold:
  - `run-on-internal-gpu.sh` (moved from top level)
  - `build-webchat.py` (new — SECRET-SAUCE preprocessor for webchat; see §4)
- `skills/` — converted skills. Initially: `ci-setup/`, `project-setup/`, `cc-log-extract/`, `callgraph/` (see §6). `cc-log-extract/` and `callgraph/` already exist (created/converted 2026-06-05, before cutover) and get picked up by the whitelist as-is.

Sensitive / secret / private:

Runtime state (auto-managed by CC):

Decided machine-local (not synced):

- `plugins/` — appears CC-managed (`blocklist.json`, `known_marketplaces.json`, `marketplaces/`)
- `00_old/` — local archive (legacy; not for brief archival anymore — see `briefs/` above)

## 3. `.gitignore` (proposed)

```gitignore
# Ignore everything at the top level by default...
/*

# ...except these tracked items:
!.gitignore
!.gitattributes
!README.md
!CLAUDE.md
!CLAUDE_webchat.md
!NEW-MACHINE-SETUP.md
!VKBASALT-SETUP.md
!settings.json
!statusline.sh
!commands
!scripts
!skills
!briefs

# Defense-in-depth: never track these even if pattern logic ever slips
.credentials.json
history.jsonl
SECRET-SAUCE.md
HARDWARE-NOTES.md
CLAUDE_webchat.expanded.md
```

**Why `/*` not `*`:** the leading slash anchors to the repo root, so when we un-ignore a directory like `commands/`, files inside it track normally without further rules.

**Tracked subdirectories track everything inside them.** If a tracked subdirectory ever needs its own exclusions, add them as targeted rules later.

**Note `BRIEF.md` not in the whitelist.** Intentional — it moves into the tracked `briefs/` directory at cutover; see §7 step 11.

## 4. SECRET-SAUCE pattern

Some material in `CLAUDE.md` and `CLAUDE_webchat.md` shouldn't be on public GitHub. It is moved into `~/.claude/SECRET-SAUCE.md` (gitignored) and referenced from each surface.

### CLAUDE.md (Claude Code)

CC supports `@path` imports inline. At an appropriate place near the bottom of `CLAUDE.md`:

```markdown
@./SECRET-SAUCE.md
```

The import is resolved by CC at load time. If the file is missing on a fresh checkout, CC should ignore the line silently — **verify this assumption on first use** (§7 step 9). If CC errors on a missing import, the pattern needs a conditional-include wrapper or pre-flight check.

### CLAUDE_webchat.md (claude.ai)

claude.ai has no `@import` mechanism — userPreferences is a plain text blob. Workflow:

1. `CLAUDE_webchat.md` (committed) contains the public template with literal `@./SECRET-SAUCE.md` placeholder lines.
2. `scripts/build-webchat.py` expands `@./SECRET-SAUCE.md` lines by inlining file contents, writes `CLAUDE_webchat.expanded.md` (gitignored), and copies to clipboard via `xclip -selection clipboard`.
3. User runs the script when needing to re-paste preferences into claude.ai.

**Why a Python script even though re-paste is rare.** In practice, the userPreferences blob lives on Anthropic's servers and is only edited via the web UI; the local copy is rarely needed. In theory, this is a raptor contingency: if the online configuration becomes inaccessible for any reason (account issue, server-side data loss, migration, etc.), the user needs to be able to reconstruct the working blob without re-doing the ~week of tuning that produced it. The script makes that reconstruction a single command rather than a manual splice-and-paste under pressure. Python is chosen for **auditability** — a small script with a top-of-file docstring explaining its purpose is easy to read and verify before running, which matters precisely because it's used in rare, possibly-stressful situations.

The script's docstring should explicitly state this rationale, so a future-self running it in an emergency knows why it exists.

**Open assumption:** the same `SECRET-SAUCE.md` content fits both surfaces. To be verified during the trim work in §5 — when the secret content is identified, decide if the same paragraph reads well in both contexts. If not, split into `SECRET-SAUCE-cc.md` and `SECRET-SAUCE-webchat.md` (both gitignored) and have each surface import its own.

## 6. Skill conversion plan

Convert three reference docs into skills so semantic triggering loads them on demand instead of keeping their content resident (or, in the third case, scoped to a single project).

### `skills/ci-setup/`

Wraps CI setup reference content. Frontmatter description draft:

> Reference for setting up or modernizing CI for a Python project in the fleet — GitHub Actions, tox, pytest coverage, cibuildwheel for Cython extensions, Windows MSVC activation, and supply-chain hardening (pinning actions to commit SHAs, least-privilege `GITHUB_TOKEN` permissions, vetting action bumps). Trigger when configuring CI workflows, adding coverage, fixing CI failures, hardening CI / pinning actions to SHAs / setting workflow token permissions, or scaffolding a new project's CI.

### `skills/project-setup/`

Wraps project setup reference content. Frontmatter description draft:

> Reference for setting up a new Python project in the fleet, or modernizing an existing one — pyproject.toml + PDM flow, Cython/meson-python editable install, lockfile policy, canonical lint/style config. Trigger when creating a new project, switching build system, or auditing an existing project's metadata.

### `skills/unpythonic-macro-testing/`

Pointer skill — semantic triggering layer over content that stays canonical in `unpythonic/CLAUDE.md`. The macro-test framework is part of unpythonic's public API, and under any selective code-preservation event it should travel with the published high-star repo, not orphan into a personal config tree. See resolution #9.

Structure:
- **Frontmatter description** carries the trigger weight. Draft:

  > Reference for writing tests for macro-enabled Python code using `unpythonic.test.fixtures` (which is the test framework unpythonic provides — name kept for backward compat despite being a slight understatement). Covers the `with session`, `with testset`, `test[]`, `test_raises[]`, `test_signals[]`, and `the[]` machinery, why test code under a macro expander needs different fixtures than ordinary pytest code, and how to structure a macro-enabled test module. Trigger when writing tests for code that uses mcpyrate macros or unpythonic macros, when debugging macro-test interactions, or when scaffolding a test module for a macro-using project.

- **Skill body** is a short overview covering the 80% case (basic test structure, the common assertion macros, the session/testset pattern) so CC can write straightforward tests without leaving the skill. Sufficient for first-pass test scaffolding.
- **Pointer to canonical reference** at the end: "Full reference: `~/Documents/koodit/unpythonic/CLAUDE.md` (search for `test.fixtures`). The path is stable across both machines by convention."

Frontmatter and overview are drafts — verify against actual `unpythonic/CLAUDE.md` content during migration.

**No deletion from unpythonic.** Asymmetric to `ci-setup` and `project-setup`, which delete their top-level sources because those notes were born in `~/.claude/` and have no preservation co-location concern.

### `skills/cc-log-extract/` (already exists)

Discoverability layer over `cc-log-extract.py` — the extractor that distills Claude Code JSONL session logs (`~/.claude/projects/<project>/*.jsonl`) into readable Markdown with model stamps. Created 2026-06-05 while building a monthly activity report; the first skill to land ahead of the cutover, so it's already on disk in `~/.claude/skills/cc-log-extract/SKILL.md`. Frontmatter triggers on "what did we build / pull the chatlogs / monthly activity report / session-log ethnography".

**Hybrid source-of-truth** — a third pattern, between the other two:

- The **tool** (`cc-log-extract.py` + its test suite) is canonical in `substrate-independent/scripts/`. The skill does **not** fork or copy it — it references the absolute path and tells a future session to bail rather than reimplement if the repo isn't cloned. Same upstream-tool discipline as `unpythonic-macro-testing`, and consistent with §8's "no restructuring of `cc-log-extract.py`."
- The **recipe** (where the logs live, the mtime-vs-content-timestamp month-boundary gotcha, group-by-project, fan-out-to-subagents synthesis for heavy months) is born in `~/.claude/` and is canonical in the skill body — personal workflow knowledge, like `ci-setup`/`project-setup`. The script's own docstring documents flags; the skill adds the procedure around it.

So: skill body lives here, tool stays upstream, nothing deleted from substrate-independent.

### `commands/callgraph.md` → `skills/callgraph/` (converted 2026-06-05)

Originally planned as a stay-a-command-but-improve-the-description item; reconsidered and **converted to a skill** instead. Rationale: callgraph is meant for the *agent* to reach for, not for the user to type `/callgraph` — and a `description:` frontmatter field is the first-class model-side triggering mechanism, whereas a command's triggering story is weaker. The old one-line description ("Generate a call graph with pyan3") was too terse to fire on prompts like "map this module's call structure"; the skill's description now enumerates trigger conditions (static call graphs, module-level dependency graphs, exploring a source tree's shape, `--text` adjacency lists for agent consumption).

Conversion is lossless: a skill is still invocable as `/callgraph` if ever typed, and gains semantic triggering. The only dropped affordance is `$ARGUMENTS` templating — unused in practice. The old `commands/callgraph.md` was deleted (single home; no duplicate "callgraph" in the skill list).

### Source-of-truth: asymmetric by content origin

Content for `ci-setup` and `project-setup` lives only in `skills/<name>/SKILL.md`. The top-level `CI-SETUP-NOTES.md` and `PROJECT-SETUP-NOTES.md` files are **deleted** after migration. These notes were born in `~/.claude/` and document personal sysadmin workflow — single source of truth in the skill dir is the clean design.

Content for `unpythonic-macro-testing` lives in `unpythonic/CLAUDE.md`; the skill is a pointer. This skill documents the public API of a published project, and the docs should travel with the code under any selective code-preservation event (see resolution #9).

The general rule: a skill's source-of-truth lives wherever the content's natural archival home is. Personal config → fleet config repo. Public API → the published repo. A skill may *split* across both — `cc-log-extract` keeps its tool upstream (substrate-independent) and its workflow recipe here — in which case each part follows the rule independently.

Discoverability for `ci-setup` and `project-setup` (whose content is in the skill dir) addressed by:
- README links into `skills/` with a brief description of what's there.
- `skills/<name>/SKILL.md` files render normally on GitHub; the path is shallow and easy to navigate.
- Skills carry semantic triggers via their frontmatter, so CC loads them on demand — no need for them to be at the top level for that purpose.

## 7. Migration sequence

All work happens in an offline copy first. Sequence:

1. **Snapshot.** `cp -a ~/.claude ~/koodit/dotclaude-staging/` (or `git clone` an empty repo into a sibling and rsync). Treat the staging copy as the working area.
2. **`.gitignore` first.** Author the whitelist `.gitignore` in the staging copy. `git init`, `git add .`, `git status` — verify only intended files are staged. Iterate until clean.
3. **Identify SECRET-SAUCE content.** Read CLAUDE.md and CLAUDE_webchat.md together. Pull private bits into `SECRET-SAUCE.md`. Replace with `@./SECRET-SAUCE.md` lines. Verify the same content fits both surfaces (or split if not — see §4).
4. **Trim CLAUDE.md.** Apply §5 plan. Move Hardware section to `HARDWARE-NOTES.md` (gitignored). Drop pyan3 paragraph from Tools section. Re-measure size.
5. **Scrub pass for public-repo safety.** Required because the GitHub repo is public. Read every tracked file (CLAUDE.md, CLAUDE_webchat.md, README.md, VKBASALT-SETUP.md, NEW-MACHINE-SETUP.md, settings.json, statusline.sh, scripts, skills, briefs) end to end and check for:
   - Real names, employer names, locations beyond what's already on the user's public GH profile
   - Internal URLs, hostnames, IP addresses, dashboard names
   - References to non-public projects, clients, or research
   - Paths under `/home/<user>/` that reveal usernames (anonymize to `~/` where it doesn't break a literal-path instruction)
   - Anything else that the user wouldn't want a stranger reading

   For each finding: either move into `SECRET-SAUCE.md` / `HARDWARE-NOTES.md`, anonymize, or remove. Surface findings to the user for sign-off before proceeding.
6. **Move scripts.** Create `scripts/`, move `run-on-internal-gpu.sh` into it, write `scripts/build-webchat.py` (with the raptor-contingency docstring per §4). Update any in-text references in CLAUDE.md (`source ~/.claude/run-on-internal-gpu.sh` → `source ~/.claude/scripts/run-on-internal-gpu.sh`).
7. **Convert skills.** Create `skills/ci-setup/SKILL.md` and `skills/project-setup/SKILL.md`, migrating content from the top-level `*-NOTES.md` files — **including `CI-SETUP-NOTES.md` §7a/§7b (supply-chain hardening); see the §5 note**. Delete the now-redundant top-level files. Create `skills/unpythonic-macro-testing/SKILL.md` as a pointer skill (rich frontmatter + 80%-case overview + reference to canonical content in unpythonic) — do **not** delete from unpythonic. `skills/cc-log-extract/SKILL.md` and `skills/callgraph/SKILL.md` already exist (created/converted 2026-06-05) — no work needed beyond confirming the whitelist tracks them. (`callgraph` was converted from the old command per §6; `commands/` is now empty.)
8. **Test the build.** Run `scripts/build-webchat.py`, verify expansion. Open a CC session pointed at the staging copy if possible, verify `@./SECRET-SAUCE.md` resolves and trimmed CLAUDE.md still drives behavior correctly. Test that CC tolerates a missing `SECRET-SAUCE.md` (simulate fresh checkout by temporarily renaming).
9. **Write README.md.** Now that the layout is verified, document what's actually there. Content per the user-audience plan below.
10. **Commit & push.** Initial commit on GitHub as a **public repo**.
11. **Cut over `~/.claude/` itself on the first machine.** Stop CC. `mv ~/.claude ~/.claude.pre-sync.bak`, clone the repo to `~/.claude`, copy `SECRET-SAUCE.md` and `HARDWARE-NOTES.md` from the backup, move the current `BRIEF.md` into `briefs/BRIEF-2026-04-30.md` and commit (this is the brief's permanent home; the design history is now part of the repo), restart CC, verify everything works. Keep `~/.claude.pre-sync.bak` for at least a week.
12. **Bring up the second machine.** Same procedure: backup, clone, manually transfer the private files (`SECRET-SAUCE.md`, `HARDWARE-NOTES.md`) from the first machine via the user's preferred private channel, verify.

### README content (for step 9)

The README needs to do real work — it's the first thing visitors see and the orientation document for a future-you on a new machine. Sections to include:

1. **What this is.** "Configuration for Claude Code (Anthropic's CLI coding agent) on the user's dev machines. Public-source-of-truth for the parts that aren't secret; designed to be cloned into `~/.claude/` on any new machine."

5. **Deploying on a new machine.** Step-by-step: clone, place SECRET-SAUCE.md and HARDWARE-NOTES.md from trusted private channel, restart CC. Cross-link to `NEW-MACHINE-SETUP.md` for the wider OS-level setup.

## Cutover addendum (2026-07-13)

Added at execution time. The body above is the brief as it stood before the migration ran; it is left verbatim, including the parts this addendum corrects.

**§4's central assumption was wrong.** The brief assumed a dangling `@./SECRET-SAUCE.md` import would be *silently ignored* by Claude Code, and left this flagged as "verify this assumption on first use." Tested empirically at cutover: the import line is **never** removed from the text. It is passed through into the model's context verbatim whether or not the target exists; when the target does exist, its content arrives as an additional, separate block rather than being spliced in at the `@` line.

The consequence is benign — a missing `SECRET-SAUCE.md` produces no error, and the session works normally, which is all the pattern actually needed. But the stated mechanism ("the line disappears") was not what happens, and the conditional-include wrapper the brief held in reserve is unnecessary. The docs do not specify the missing-file behavior either way; this was determined by running it.

**CLAUDE.md was 28 KB at cutover, not the 19.6 KB §5 measured** — it grew in the intervening ten weeks. The §5 target of ~11 KB was therefore not met and was not pursued: the growth is load-bearing rules added deliberately since, and §5's own "stays resident (verbatim)" list already covers most of the file. The moves §5 specified were all performed; the resulting file is ~22 KB.
